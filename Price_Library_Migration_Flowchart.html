Price Library Migration — Process Flow

WDS → P21 pricing migration lifecycle — from legacy data through import readiness
CAVINESS numbers annotated as reference example
Phase 1 — Source Data
Customer's WDS Price Categories
Every price category the customer has in the legacy WDS system. Each one carries a factor (the markup percentage WDS used) that gets converted into a multiplier for P21.
CAVINESS: 311 categories in the Price Matrix
matched against
Hantover Category Map
The master translation table mapping every known WDS price category to a P21 product group. Built once, shared across all 39 customer libraries — the backbone of the migration.
CAVINESS: 341 rows (includes 52 CAVINESS-specific mappings + shared Hantover mappings)
each category is looked up
Phase 2 — Mapping & Assignment
Does the category have a direct match in the map?
The system looks up each WDS category by its code. If it finds a product group assignment, we have a match. If not, we try looking at the parent category — the broader grouping it falls under.
three possible outcomes
Direct Match
The category code was found in the map and pointed to a specific product group. Clean, unambiguous.
278 categories matched directly
No Direct Match — Try Parent
The category wasn't in the map, so the system checks its parent. If the parent has a clean assignment and the children share the same rate, they inherit it.
49 categories resolved via parent inheritance (Pattern A)
No Match — Dead End
Neither the category nor any parent produced a usable product group. No automated assignment is possible.
14 categories — flagged as MANUAL
Phase 3 — Confidence & Conflict Check
How confident is the assignment?
Just because a category found a product group doesn't mean the match is bulletproof. A confidence test is applied: "Would a reasonably informed person consider this assignment obvious?" If yes, it's READY. If there's ambiguity, it's flagged SOFT REVIEW for someone with pricing knowledge to confirm.
Do multiple categories land in the same product group at different rates?
Even when the product group is correct, a conflict arises if several legacy categories all map to the same group but carried different markup rates in WDS. P21 only allows one rate per product group per customer — so someone has to pick which rate survives.
produces the page template
Phase 4 — Page Template Rows (the Deliverable)
Each resolved category generates a row on the Page Template
One row per unique product group + multiplier combination. If a product group has 3 competing rates, it produces 3 rows. Each row is color-coded and flagged based on the checks above.
CAVINESS: 102 distinct product groups → 184 Page Template rows (inflated by multi-rate groups)
six possible row statuses
✓ Ready
Clean match, single rate. Imports as-is. No action needed.
~60 rows
✓ Ready (Pattern A)
Inherited from parent category. Single rate, no conflicts. Same outcome — imports cleanly.
⚡ Soft Review
Product group was assigned but the match was ambiguous. Needs someone with pricing knowledge to confirm or redirect.
1 row
⚡ Multiplier Conflict
Product group is correct, but multiple legacy rates compete. Someone picks which rate P21 keeps.
105 rows (across 41 product groups)
⚡ Soft + Conflict
Both issues stacked. Confirm the product group AND pick a rate.
18 rows
❓ NOCAT + Soft Review
No meaningful product group could be identified — typically broad vendor umbrella categories spanning too many product types. NOCAT is a placeholder meaning "uncategorized." Someone needs to either accept that or assign a real group.
✗ Manual
No mapping exists by any method. These categories don't have a Page Template row at all. Someone must manually assign a product group before these products have pricing in P21.
14 categories (not on the Page Template)
Phase 5 — Review & Resolution
Sales / Pricing Leadership Reviews Yellow Rows
The people who understand the customer relationship open the workbook and work through the yellow rows on the Page Template. The Legacy Price Categories column shows which WDS categories feed each row — so reviewers can trace back to what they already know. The Notes column explains exactly what decision is needed.
three types of decisions
Rate Selection
For multiplier conflicts: choose which markup rate P21 should use. The Notes column shows all competing rates and how many legacy categories use each one. Options include lowest rate (most conservative for the customer), highest (best margin), most common (least disruption), or weighted average.
This is the single biggest unlocker — one policy decision resolves ~60% of yellow rows across all 39 libraries.
Group Confirmation
For Soft Review and NOCAT items: confirm the assigned product group is correct, or reassign to a different one. These are judgment calls where the data pointed in a direction but didn't give a definitive answer.
Manual Assignment
For the 14 blocked categories: assign a product group from scratch. Once assigned, a new Page Template row is created. This workstream runs in parallel with the yellow-row decisions — it doesn't depend on the multiplier strategy.
after each decision
Clear the Note → Row Flips to Ready
Once the reviewer makes the call, they delete the text in the Notes column for that row. The LEGEND sheet's progress tracker reads the Notes column with live formulas — clearing a note automatically shifts that row from "needs decision" to "ready to import." No manual count updates needed.
Source of truth: the Notes column drives the tracker, not the row color. If they ever disagree, the note wins.
Phase 6 — Collapse & Import
Collapse to One Row Per Product Group
Once all multiplier conflicts are resolved, the duplicate rows (from competing rates) are removed. Each product group gets exactly one row with the chosen rate. The Page Template shrinks from its inflated state down to one-row-per-group — which is what P21's import requires.
CAVINESS: 184 rows → 102 rows (one per product group)
all rows green, all notes cleared
Import-Ready for P21
The Page Template is clean — one row per product group, one rate per row, no flags, no notes. The 135-column P21 import schema is ready to load into Prophet21. The customer's pricing goes live on May 4th.
